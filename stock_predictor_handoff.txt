FRED MACRO DASHBOARD
--------------------
/api/macro-dashboard fetches 11 FRED series in parallel via ThreadPoolExecutor(max_workers=6).

FRED API call format (FIXED May 4 2026 — do not revert):
  Monthly series (most indicators):
    ?series_id=CPIAUCSL&limit=26&sort_order=desc&api_key=<KEY>&file_type=json
    (NO frequency or aggregation_method — adding them causes HTTP 500 on FRED)

  Daily series only (DGS10, DGS2 — marked "daily": True in FRED_INDICATOR_META):
    ?series_id=DGS10&frequency=m&aggregation_method=avg&limit=26&sort_order=desc&api_key=<KEY>&file_type=json

  _fred_obs(series_id, limit, daily=False) — daily flag controls which format is used.
  NEVER use aggregation_method=eop — FRED returns 500 for it on CPILFESL, FEDFUNDS, UMCSENT, CFNAI.

Uses urllib.request (stdlib only) — no external library needed.

11 indicators:
  CPI        CPIAUCSL   %    yoy    good=down    monthly (no freq override)

SECURITY HARDENING (applied May 4 2026 — do NOT revert)
---------------------------------------------------------
Six vulnerabilities were found and fixed. If deploying from an older api.py or models.py,
re-apply all of these.

models.py:
  - market_id whitelist: only "us", "tase", "sp500", "nasdaq100" accepted (field_validator).
    Anything else → HTTP 422 before hitting DB or spawning a thread.
  - top_n clamped: ge=1, le=50. Was unbounded — attacker could request 1,000,000 results.
  - min_confidence clamped: ge=0.0, le=1.0. Was unclamped — negative value bypassed filters.
  - task_id format enforced: alphanumeric/dash/underscore, max 64 chars (_TASK_ID_RE).

api.py:
  - Rate limit now guards ALL scan starts (not just force_refresh=True).
    Old bug: cold-cache + force_refresh=False had zero rate limiting → unlimited thread spawn.
    Fix: rate limit check moved to run before any background scan, regardless of force_refresh.
  - Concurrent scan cap: _SCAN_SEMAPHORE = threading.Semaphore(3).
    Max 3 simultaneous full market scans. 4th gets "Server busy" error, semaphore released in finally.
  - GLOBAL_PROGRESS memory leak fixed: entries now popped from dict when results are consumed
    in get_scan_progress(). Previously grew forever under sustained use.
  - _scan_rate_limit memory leak fixed: entries older than SCAN_COOLDOWN * 2 pruned on each
    scan request. Previously grew with every unique IP that ever hit the endpoint.
  - task_id probe attack fixed: GET /api/scan/progress/{task_id} now validates format with
    TASK_ID_RE before any dict lookup. Returns HTTP 400 on invalid format.
  - Removed duplicate `import threading` from inside scan_market() — already imported at module level.

OPTIONS LIQUIDITY FIX (May 2 2026)
-------------------------------------
fetch_options_features() previously used ATM-only OI (3 strikes) as liquidity check.
Large-cap stocks concentrate put OI far OTM (portfolio hedges) → ATM OI < 100 → null options.

Fix: liquidity check now uses TOTAL put/call OI ≥ 200 per side (not ATM-only).
PC ratio: volume-based (≥50 contracts per strike), falls back to OI if volume too low.
This affects ALL stocks, not just WDC.

         Verify: check first lines of pre_scan.py load api_data.env
Cause B: FRED API returned HTTP 500 (happens occasionally)

         Verify: run debug_fred.py on server

Symptom: Scanner results unchanged for days
Cause:   pre_scan.py silent failure — check pre_scan.log tail
Fix:     Check last "Pre-scan complete." timestamp; Telegram alert should arrive daily at ~05:05

Symptom: Options data null / pc_ratio missing for a large-cap stock
Cause A: Old bug (pre May 2 2026) — ATM-only OI check too strict; already fixed
Cause B: Stock genuinely has < 200 total OI per side → truly illiquid options
Fix:     Check Yahoo Finance options chain; if liquid, check fetch_options_features() liquidity threshold

Symptom: Scanner returns 0 BUY results (or very few) even though market is up
Cause A: SCAN_CONFIDENCE_THRESHOLD too high relative to light_mode calibration.
         Light mode (100 iter) produces ~0.10 lower confidence than full mode (300 iter).
         If threshold is 0.67, stocks that would score 0.70+ in full mode score ~0.57-0.60 in light.
         Fix: SCAN_CONFIDENCE_THRESHOLD = 0.57 (set May 13 2026)
Cause B: MIN_PRECISION_BUY too high during post-correction regime.
         Test set (last 20% of 5y data) includes correction period → BUY precision scores
         temporarily suppressed (0.30-0.36 instead of 0.48+) because many signals failed in decline.
         Fix: lower MIN_PRECISION_BUY temporarily (0.30 as of May 13 2026); restore ~Nov 2026.
Cause C: CMF filter blocking signals — broad money outflow after correction.
         Fix: check df["cmf"].iloc[-1] for candidate stocks; if mostly negative, regime issue.
Cause D: FRED macro features missing/NaN → model predictions degrade.
         Fix: run debug_fred.py on server to verify macro join is healthy.

Symptom: NASDAQ100 pre_scan shows same signal count as SP500 (e.g. 483 vs 483)
Cause:   load_sp500() and load_nasdaq100() shared same TTLCache object — cache key collision on ()
Fix:     Ensure each function uses its own TTLCache: _sp500_cache, _nasdaq_cache (fixed May 5 2026)
         If this reappears after deploying old core_logic.py, re-apply the separate cache instances.

Symptom: BUY signal dropped from scanner but appears in individual predict
Cause:   Options filter penalized confidence below 0.70 in batch scan.
         Individual predict retrains fresh → slightly different confidence → may cross threshold.
Fix:     Use ALMOST BUY tab in scanner → click "סרוק פרטנית" to confirm via detail scan

Symptom: FRED dashboard shows None for CORE_CPI, FED_FUNDS, CFNAI, M2, or SENTIMENT
Cause:   _fred_obs() was using aggregation_method=eop — FRED returns HTTP 500 for this on monthly series.
         If deploying from an old api.py (pre May 4 2026), this bug will reappear.
Fix:     Ensure _fred_obs() only uses frequency=m&aggregation_method=avg for DGS10/DGS2 (daily=True).
         All other series must NOT pass frequency or aggregation_method. See FRED MACRO DASHBOARD section.

Symptom: Same ticker returns different signal on two consecutive calls (e.g. BUY 79% then HOLD 53%)
Cause:   FRED data availability — t10y2y is a top-5 feature. With FRED available: real values.
         With FRED down (fallback ^TNX-^IRX): different t10y2y → signal can flip entirely.
         Also: core_logic.py caches fetch_stock_data 15min — a fresh Python process gets new data
         while the running API serves the cached result.
Fix:     Run debug_fred.py on server to check FRED status.
         API result is authoritative (uses cached FRED from last successful fetch).

Symptom: Frontend JS/CSS 404 after deploy
Cause:   Only index.html was copied to frontend/dist/ — Vite hashes asset filenames each build.
Fix:     Always cp -r the full dist/ to frontend/dist/ (broke site May 10 2026).

CHAOS FILTER (meme/high-vol stock exclusion)
---------------------------------------------
Applied in both _train_single() (scanner) and get_prediction() (single lookup).
Rule: avg ATR over last 30 trading days > chaos_threshold → skip
  chaos_threshold is VIX-based (dynamic):
    VIX < 15  → 5%
    VIX < 25  → 7%
    VIX >= 25 → 10%   (high-vol regime: legitimate stocks spike ATR; looser threshold needed)
  - Scanner: silently returns None
  - Single lookup: returns signal="EXCLUDED" with excluded_reason field
Rationale: 10-day swing model is structurally wrong fit for meme stocks (HOOD, GME, etc.)
The dynamic threshold was added because during corrections (VIX>25), many legitimate stocks
temporarily exceed 7% ATR and would be excluded incorrectly without regime-aware thresholding.

Purpose:
  High-volume stocks often signal momentum opportunities. This tab finds them daily,
  filters noise (mkt cap < $200M), and layers the existing ML model on top.

Backend endpoint: GET /api/volume-leaders?min_market_cap=200000000&force=false
  File: api.py — function get_volume_leaders()
  Cache: 30 min (_volume_leaders_cache). Pass ?force=true to bypass.

Data pipeline:
  1. Yahoo Finance unofficial screener API (query1.finance.yahoo.com) — top 50 most active
  2. Filter: marketCap >= $200M → keeps up to 20 stocks
  3. ML signal: query scanner_cache.db for today's sp500 + nasdaq100 pre-scan results
     (2 queries total, dict lookup — no N+1 loop)
  4. yfinance batch download (1mo daily) → RSI(14) + 5-day return per ticker
  5. Volume ratio: regularMarketVolume / averageDailyVolume3Month (from screener data)

Verdict logic (_compute_verdict in api.py):
  OVEREXTENDED — price up ≥3% from open (anti-chasing guard)
  VOL BREAKOUT — vol_ratio ≥ 3x + RSI < 65 (independent of ML)
  BUY          — ML=BUY + conf ≥ 0.65 (if vol_ratio ≥ 2x) or ≥ 0.70 (normal)
  HOLD         — everything else incl. ML=SELL (SELL suppressed — Long-only Expert)
  WATCH        — RSI < 35 + vol_ratio ≥ 2x (momentum only, no ML)

  NOTE: ML=SELL → HOLD (SELL suppressed since May 12 2026 — Long-only Expert)

Response format (wrapped object — NOT a plain array):
  { "results": [...], "fetched_at": "2026-05-10T18:32:00Z" }
  Frontend uses json.results and json.fetched_at separately.

Frontend component: VolumeLeadersView() at end of App.jsx
  Table columns: Ticker | Name | Price | Mkt Cap | Volume | Vol/Avg | 5d Ret | RSI | ML Signal | Verdict
  Vol/Avg highlight: ≥ 3x → orange bold + 🔥, ≥ 2x → yellow, < 2x → gray

  Refresh button: calls ?force=true to bypass 30-min cache

VOLUME LEADERS — ON-DEMAND ML (added May 10 2026)
---------------------------------------------------
Enhancement to the Volume Leaders tab: stocks outside S&P 500 / NASDAQ 100
(e.g. IREN, NOK, RKLB) now receive a live ML prediction instead of showing N/A.

How it works:
  After loading sp500_map + nasdaq_map from scanner_cache.db, collect uncached tickers.
  Run core_logic.get_prediction(sym, light_mode=True) on all of them in parallel via
  ThreadPoolExecutor(max_workers=5). One failure per stock is isolated — won't crash
  the endpoint. Results merge into live_predictions dict.

  In the per-stock loop:
    ml = sp500_map.get(sym) or nasdaq_map.get(sym)
    ml_live = False
    if not ml and sym in live_predictions:
        ml = live_predictions[sym]
        ml_live = True

  Result dict includes "ml_live": True/False.
  "EXCLUDED" signal (chaos filter: ATR too high) is treated as N/A.

  light_mode=True: matches the scanner (max_iter=100, faster, same accuracy).
  ThreadPoolExecutor is already imported at top of api.py.

Frontend:
  ML Signal cell: shows ⚡ next to signal when ml_live=True.
  ⚡ has title tooltip: "Signal computed live — stock is outside the regular
  S&P 500 / NASDAQ 100 watchlist"
  Bottom legend: "⚡ = signal computed live on-demand — stock is outside
  the regular S&P 500 / NASDAQ 100 watchlist."

Performance:
  ~2-5 sec added to first load if uncached stocks exist.
  30-min endpoint cache means this runs once per half-hour, not per user.
